User self-describes as less practiced on the frontend side than on backend work. They've asked the agent to proactively explain *why* behind frontend decisions so they can learn the rule and apply it later.

**How to apply:**
- Maintain the staff-level bar on what the code *is*. The adjustment is in how you *explain*, not in what you ship.
- When picking an approach, briefly state why in plain terms (one line is usually enough).
- Flag the non-obvious explicitly. If a decision looks weird (e.g. "we're storing filters in the URL, not in state"), name it and give the rule so the user can apply it next time.
- Don't assume the user will push back on a bad suggestion — they may not know to. Be your own skeptic first.
- Re-verify when the user signals their frontend skill has grown or their role has shifted; soften the proactive-explanation tone accordingly.
